================================================================
    牌效分析算法优化方案
    当前性能: 总耗时约 2.3 秒（进张分析 4ms + 好型率 1700ms）
================================================================

【性能瓶颈分析】
────────────────────────────────────────────────────────────────

实测数据（手牌 1m4m4m7m9m 3p4p4p6p7p 1s6s7s8s）：

  单次向听计算:     0.026ms
  进张分析:         4.6ms    （5个切牌选项 × 34种进张检查）
  好型率（单张）:   577ms
  好型率（全部）:   1707ms   （占总耗时 99.7%）

瓶颈定位：好型率计算中的递归搜索

  calculate_good_shape_rate() 递归深度3层：
  - 深度0: 模拟34种摸牌 × 切牌组合
  - 深度1: 每种组合再模拟34种摸牌 × 切牌
  - 深度2: 再递归一层
  - 每层都需要调用 calc_shanten()（递归搜索算法）
  
  预估调用次数: 34 × 14 × 34 × 14 × ... ≈ 数万次向听计算


================================================================
方案一：缓存优化（推荐指数：★★★★★）
================================================================

原理：
  当前每次调用 calc_shanten_cached() 都用新的空字典作为缓存，
  导致相同的牌型组合被重复计算。

优化方式：
  1. 在 evaluate_hand() 中创建一个全局 shanten_cache
  2. 将该 cache 传递给所有 calculate_good_shape_for_discard() 调用
  3. 好型率内部也复用同一个 cache

代码改动：
  evaluate_hand() 函数中：
  - shanten_cache = {}  ← 提升为全局共享
  - calculate_good_shape_for_discard(hand, tile, shanten_cache)  ← 传入cache
  
  calculate_good_shape_for_discard() 中：
  - 不再创建新的 shanten_cache
  - 使用传入的 cache

预期效果：
  向听计算结果可被跨切牌选项复用
  预计减少 60-80% 的向听计算调用
  耗时: 1700ms → 400-600ms

利弊：
  + 改动极小（约5行代码）
  + 不影响算法正确性
  + 效果显著
  - 不能从根本上解决递归爆炸问题


================================================================
方案二：限制好型率计算切牌选项（推荐指数：★★★★☆）
================================================================

原理：
  当前对所有5个切牌选项都计算好型率，
  但实际上用户只关心进张数最多的前2-3个选项。

优化方式：
  只对进张数排名前2的切牌选项计算好型率
  其他选项的好型率显示为 null 或 "-"

代码改动：
  analyze_hand_efficiency() 中：
  - 对 efficiency 排序后，只对前2个调用 calculate_good_shape_for_discard()
  - 其他选项的 good_shape_rate 设为 None

预期效果：
  好型率计算次数: 5次 → 2次
  耗时: 1700ms → 700ms

利弊：
  + 改动极小
  + 用户体验影响小（最重要的选项都有好型率）
  - 被跳过的选项无法显示好型率


================================================================
方案三：限制递归深度（推荐指数：★★★☆☆）
================================================================

原理：
  当前 MAX_GOOD_SHAPE_DEPTH = 3，
  每增加一层深度，计算量指数级增长。

优化方式：
  将 MAX_GOOD_SHAPE_DEPTH 从 3 降为 2

代码改动：
  shanten.py 中：
  - MAX_GOOD_SHAPE_DEPTH = 2

预期效果：
  递归层数减少一层
  耗时: 1700ms → 400-500ms

利弊：
  + 改动极小（1行）
  - 好型率精度降低（深度不足可能导致结果偏差）
  - 对某些手牌可能返回 0（无法在有限深度内找到好型路径）


================================================================
方案四：异步计算 + 进度反馈（推荐指数：★★★★☆）
================================================================

原理：
  不优化算法本身，而是优化用户体验。
  将好型率计算放到后台线程，前端实时显示进度。

优化方式：
  1. 后端API先返回进张分析结果（4ms，秒回）
  2. 前端立即显示进张数据
  3. 后端异步计算好型率，通过轮询或SSE推送更新
  4. 前端收到后补充显示好型率

代码改动：
  - 后端：新增 /api/calculate-good-shape 异步接口
  - 前端：先显示进张数据，再异步获取好型率

预期效果：
  首屏显示时间: 2300ms → 50ms
  好型率延迟显示: 约2秒后出现

利弊：
  + 用户体验最好（秒出进张数据）
  + 不影响算法精度
  - 改动较大（需要新增API + 前端异步逻辑）
  - 需要处理并发和状态管理


================================================================
方案五：剪枝优化（推荐指数：★★★★☆）
================================================================

原理：
  在好型率递归中，如果某个分支的当前向听数已经比
  目标向听数大很多，可以提前剪枝。

优化方式：
  在 calculate_good_shape_rate() 中：
  - 如果 depth >= 1 且当前向听数 > 2，直接返回 0
  - 如果摸牌后向听数没有下降，跳过该分支

代码改动：
  calculate_good_shape_rate() 函数中：
  - 添加 early exit 条件

预期效果：
  减少无效递归分支
  耗时: 1700ms → 800-1000ms

利弊：
  + 改动适中
  + 不影响核心算法
  - 可能导致部分手牌的好型率偏低（被剪枝的分支可能是好型）


================================================================
方案六：批量向听计算（推荐指数：★★★☆☆）
================================================================

原理：
  当前逐个计算34种进张的向听数，
  可以批量计算，减少函数调用开销。

优化方式：
  新增 calc_shanten_batch(tiles34_list) 函数
  一次性计算多个牌型的向听数

代码改动：
  - 新增批量计算函数
  - evaluate_hand() 中使用批量计算

预期效果：
  减少Python函数调用开销
  耗时: 1700ms → 1200-1400ms

利弊：
  + 对上层透明
  - 改动较大
  - 效果有限（主要瓶颈是递归次数，不是单次计算）


================================================================
推荐组合方案
================================================================

最佳实践：方案一 + 方案二 + 方案五

  1. 全局共享 shanten_cache（方案一）
     → 减少 60-80% 向听计算
  
  2. 只对前2个选项计算好型率（方案二）
     → 减少 60% 好型率计算
  
  3. 递归剪枝（方案五）
     → 减少 40-50% 递归分支

组合预期效果：
  当前: 2300ms
  优化后: 200-400ms
  
  提速约 6-10 倍
